# 「背了吗」 AI 学习助手 · 开发纪要

> 本文档记录 card-back 内嵌 AI 聊天机器人（兔叽咪）从技术评估到落地、再到上下文治理的完整过程。
> 重点不在“写了什么代码”，而在 **每个决策背后的工程化取舍**，供后续迭代和同类 Agent 功能参考。

---

## 0. 功能概述

- 位置：学习页翻卡看到答案（card-back）时，右下角悬浮图标，点击弹出对话面板。
- 记忆策略：整个学习队列共享一份聊天记录（策略 A），退出队列自动销毁。
- 模型：Kimi (Moonshot)，OpenAI 兼容协议，SSE 流式输出。
- 涉及文件：
  - `src/utils/kimi.ts` —— API 封装、System Prompt、卡片上下文、token 估算与截断
  - `src/components/learn/AIChatBot.tsx` —— 聊天组件（UI + 消息编排）
  - `src/components/LearnPage.tsx` —— 挂载点，持有队列级聊天记录
  - `src/App.css` —— 聊天 UI 样式（末尾 `AIChatBot` 段）

## 1. 方案评估：先对齐现实，再动手

拿到一份“完整技术方案”后没有直接照抄，而是**先用现有代码校验方案的每个前提**。这一步避免了大量返工。

| 方案假设 | 实际情况 | 结论 |
| --- | --- | --- |
| `components/learn/LearnPage.tsx` | 实际是 `components/LearnPage.tsx` | 修正引用路径 |
| `onMessagesChange: (msgs[]) => void` | 代码里却用了函数式更新 `prev => ...` | 类型与实现自相矛盾，`tsc` 不过 |
| 流式回调基于快照 `nextMessages.map` | 每个 chunk 都基于初始空快照拼接 | 只会显示最后一个片段（覆盖而非追加），必须函数式更新 |

**工程化要点**
- **不迷信“看起来完整”的方案**：外部方案的技术栈假设、文件结构、类型定义都要与真实工程对齐。
- **提前识别阻断性问题**（编译不过 / 运行异常）与建议性问题（体验优化），分级处理。
- 复用已有能力而非重造：项目已内置 `marked` + `highlight.js`，AI 回复直接走 Markdown 渲染，无需另造纯文本方案。

## 2. 定位陷阱：`position: fixed` 为何“悬浮不起来”

**现象**：机器人按钮随卡片内容滚动；对话面板与答案“同层级”，不像弹窗。

**根因**：`.card-back` 上有 `transform: scale()` 和 `overflow-y: auto`。CSS 规范中，**只要祖先设置了 `transform`，其后代的 `position: fixed` 就会退化为相对该祖先定位**，于是 fixed 元素被“困”在卡片内、跟着滚动。

**解决**：用 `createPortal` 把整个 AIChatBot 渲染到 `document.body`，彻底脱离 `.card-back` 的 transform / overflow 上下文。

```
// AIChatBot.tsx
return createPortal(<> ...按钮 + 面板... </>, document.body)
```

- React 组件树中 `<AIChatBot>` 仍挂在 card-back 内（随 `flipped` 挂载/卸载），但实际 DOM 渲染到 body。
- fixed 定位恢复相对视口生效；面板 z-index 高于所有内容，成为真正的顶层弹窗。

**工程化要点**
- 遇到“样式写对了却不生效”，优先排查**层叠上下文 / 定位上下文**这类 CSS 隐式规则，而非反复调数值。
- Portal 是解耦“逻辑归属”与“渲染位置”的标准手段，适用于弹窗、悬浮层、Toast。

## 3. 数据来源：不该查的云端不要查

**需求**：三个快捷按钮（举个例子 / 相关题目 / 学习建议）需要读到当前题目和答案，而“题目和答案都在云端数据库”。

**关键判断**：题库在应用启动时已由 `api.ts` 整体拉取到前端内存，`Question` 类型本就包含 `q / a / summary / pitfalls / interview / cat` 全字段。学习页手上的 `card` 就是完整对象——**无需任何额外网络请求**。

之前的 bug 是只把 `card.q`（题目）传给了机器人，没传答案，导致 AI 在“盲答”。修复：把完整 `card` 作为 `currentCard` 传入，注入上下文。

**工程化要点**
- **先厘清数据已在何处**，再决定是否发起请求。能用内存数据就不查库，省延迟、省成本、省复杂度。
- “数据在云端”不等于“每次都要访问云端”；缓存 / 预加载后的内存副本同样是可信数据源。

## 4. 核心难题：上下文错乱、膨胀与幻觉

这是 Agent 开发的真正深水区。三个问题看似独立，实则都根源于 **“发给模型的 messages” 与 “界面显示的历史” 没有正确治理**。

### 4.1 问题一 —— 记错题（上下文漂移）

**现象**：连问两题后，问“第一题是什么”，模型答成了第二题。

**根因**：卡片信息被塞进 **System Prompt**，而 System Prompt 每次请求都用 **“当前卡片”重新生成**。翻到第二题后，System 层反复强调“当前题目是第二题的答案”，权重极高，污染并覆盖了真实的历史顺序。**本质是上下文动态漂移，不是模型记忆力问题。**

**解决 —— 分层消息结构**

| 层级 | 内容 | 特点 |
| --- | --- | --- |
| System（固定） | AI 身份、风格、防幻觉边界 | 整个会话只设一次，**永不漂移** |
| 卡片背景（随卡片） | 当前题目 + 标准答案 | 作为一条**真实历史消息**，在“切卡后首次提问那一刻”插入，写入后**固化不改** |
| 对话历史 | 用户与 AI 的真实往返 | 严格时间顺序 |

落地关键（决定成败）：
- 卡片背景**不是每次发送时动态拼装**，而是**写入持久 `messages` 数组的一条真实记录**，一旦写入就钉死在该位置，翻卡只往后追加、绝不回改。
- 使用 `apiRole: 'system'`（中途系统提示）+ `hidden: true`（界面不渲染，只发给模型）。
- **同卡只插一次**：用 `lastInjectedQRef` 记录上次注入的题干，同卡连续提问不重复插，省 token。

于是模型看到的是严格时间线：
```
[System] 身份+约束
[system·hidden] 卡A背景
[user] 问题1
[assistant] 回答1
[system·hidden] 卡B背景
[user] 问题2
[assistant] 回答2
[user] 还记得第一题吗？   → 第一题=A 永久钉死，不会串题
```

### 4.2 问题二 —— 上下文膨胀（记忆管理与压缩）

队列共享一份对话，题目一多，messages 无限增长 → 触发上下文超限、费用飙升、响应变慢。

**压缩时机（关键认知）**
- 压缩发生在**每次发送前、组装 apiMessages 的那一刻**，不是收到回复后。
- **界面 `messages` 永远全量保留**（用户可翻完整历史），**只压缩即将发给 API 的那一份**。两者分离。
- 只压最老的部分，**最近若干轮必须留原文**（追问依赖近期精确内容）。

**本次落地：滑动窗口 + token 预算**（`buildApiMessages`）
- System Prompt 置顶，永不截断。
- 卡片背景（system 消息）受保护、保持原位不丢（否则模型忘了早期题目是什么）。
- 普通对话从最老开始丢弃，直到落入 `TOKEN_BUDGET`；无论如何至少保留最近 `KEEP_RECENT` 条。
- **全程保序**，保证“第几题”的时间线正确。
- token 估算用经验值（中文≈1/字，其余≈4字符/token），前端无需引入 tokenizer。

**未来可选：摘要压缩**（暂未实现）
- 达到软阈值时，把被丢弃的老对话用一次 LLM 调用总结成一段摘要，拼在 System 后。
- 摘要结果需**缓存复用**，不可每轮重算（否则费用与延迟不可控）。
- 分阶段落地：先滑动窗口覆盖 90% 场景，量真的大了再上摘要。

### 4.3 问题三 —— 模型幻觉

本场景有权威知识源（题库标准答案），属于典型可控幻觉场景。手段：

1. **强化 System 约束**：优先依据标准答案；答案未涵盖需明确声明“非标准答案”；禁止编造 API/参数/版本号/链接；不确定就说“我不确定”；按顺序区分第几题。
2. **RAG 思路（天然具备）**：卡片答案即“检索到的权威上下文”，让模型基于它展开而非自由发挥。
3. **降低 temperature**：`0.7 → 0.3`，输出更稳定、更贴答案。
4. **区分题内/题外**：题内严格依答案，题外可自由答但提示“为通用知识”。

## 5. SSE 流式输出的可靠性治理

流式（SSE）是 Agent 交互的标配，但“边收边渲染”的异步特性带来一系列可靠性与生命周期问题。这里把踩过的坑与解法系统化。

### 5.1 先分清两类“不完整”

| 层次 | 含义 | 谁负责 | 说明 |
| --- | --- | --- | --- |
| 传输层完整性 | 一个 SSE 事件被 TCP 拆到多次 `read()` | 解析代码 | 需正确重组 |
| 应用层完整性 | 流中途断开、模型没说完就停 | 需额外机制 | 需显式校验 |

**关键认知**：SSE 跑在 TCP/HTTPS 上，**字节流天然有序、不丢、不重**。所以“保证有序完整”**不是防丢包**（TCP 已保证），而是 **① 正确把字节流重组成事件；② 显式判断流到底走完没有**。误以为要“防网络丢包”是常见误区。

### 5.2 三道防线（`streamChat`）

**防线 1 · 字节层：不产生乱码、不丢残留**
- `TextDecoder.decode(value, { stream: true })`：多字节字符（中文占 3 字节）被 TCP 拆到两次 `read()` 时，不完整字节留到下次，杜绝半个汉字变 `�`。
- 循环结束后再 `decode()`（不带 stream）**flush** 一次，取出解码器内部残留字节。

**防线 2 · 事件层：不拆坏事件、不漏最后一行**
- buffer 累积，按换行切分，**最后一段永远留在 buffer**（可能是下个包才补齐的半行）。
- 兼容 `\r\n | \r | \n` 三种换行（SSE 规范允许）。
- while 循环结束后，**处理 buffer 残留**（防服务端最后一个 data 行不带换行符被丢弃）。
- data 行 JSON 解析失败**不再静默吞掉**，抛“片段损坏”错误；空行 / `:` 注释行正常跳过。

**防线 3 · 语义层：显式校验“真的完整结束了”**
- 只有收到 `data: [DONE]` 才算**正常完整**。
- `reader` 提前 `done` 但从未收到 `[DONE]` → 判定**异常截断**，抛错，**绝不静默当成成功**（这是最隐蔽的 bug：截断了却装完整）。
- 检查 `finish_reason`，`streamChat` 返回结束原因 `'done' | 'stop' | 'length' | 'content_filter'`，其中 `length`（撞 `max_tokens`）也是“没说完”。

### 5.3 断流补救：无法重传，只能保留 + 重来

流式内容**无法重传中间片段**（模型一次性生成，断了就没了）。补救策略：
- **保留已生成的半截内容**，末尾追加“（回答生成中断）”，绝不粗暴替换成“请求失败”而丢掉用户已读部分。
- **撞 `max_tokens`（`length`）** 时追加“回答较长已截断”提示。
- 失败 / 截断的 assistant 消息打 `error: true` 标记，气泡下方渲染 **“重新生成”** 按钮。

**重新生成的正确实现**（`regenerate`）：
- 把要重生成的回答抽成共用的流式核心 `runStream(asstId, apiMessages)`，首发与重生成复用同一逻辑。
- 取该回答**之前**的历史（不含它自己）重建 apiMessages；丢弃它**之后**的所有消息；把它重置为同 id、同位置的 loading 占位，再走 `runStream`。
- 生成中禁用按钮防重复触发。

### 5.4 流式渲染的生命周期坑：关闭面板后“输出像是中断”

**现象**：生成中关闭对话框再打开，输出偶尔像中断、偶尔又完整。

**根因（是视觉问题，不是数据问题）**：
- 关闭对话框是 `setOpen(false)`，只是 `{open && 面板}` 条件渲染消失，**组件没卸载、流仍在后台正常写入 `messages`**，数据始终完整。
- 但滚动到底的副作用依赖 `bottomRef.scrollIntoView`，关闭期间面板 DOM 卸载、`bottomRef.current` 为 `null`，这段时间的 chunk 触发滚动被跳过；重开后视图**停在旧位置**，看起来像“断了”。随机性来自关/开的时机是否卡在流中间。

**解法**：
- 改用**滚动容器 ref**（`bodyRef` → `.aichat-body`）配合 `scrollTo({ top: scrollHeight })`，不用 `scrollIntoView`（后者会影响外层页面 / Portal 场景），并判空跳过。
- 新消息滚动**仅在面板打开时**执行。
- **重开面板时**用 `requestAnimationFrame` 等 DOM 重建后，以 `behavior: 'auto'`（瞬时）强制对齐到最新内容，避免停在中途、也避免 smooth 动画在刚挂载时被打断。

**工程化要点**
- 异步流的回调要按**稳定 id 做函数式更新**（`prev => prev.map(id 命中)`），这样即便 UI 隐藏 / 重排，状态更新依旧正确落到目标消息。
- **区分“组件卸载”与“UI 隐藏”**：卸载才 abort 流（翻卡 / 退出学习页）；单纯关面板不应中断请求。
- 排查“中断”类问题先分清**数据层 vs 渲染层**：让它自然跑完看最终内容是否完整，能快速定位是视觉还是真丢数据。

## 6. 贯穿始终的工程化原则

1. **先验证前提，再写代码**：外部方案与真实工程对齐，阻断性问题优先。
2. **面向协议 / 抽象封装**：供应商、模型可替换，改动收敛到常量。
3. **分层隔离排障**：API 层（curl）→ 环境层（env）→ 前端层（Console/Network），逐层定位；“中断”问题先分清数据层 vs 渲染层。
4. **UI 显示与模型输入分离**：界面全量、可读；API 输入经治理（注入、隐藏、截断）。这是 Agent 上下文管理的核心心智模型。
5. **数据来源最短路径**：能用内存就不查库。
6. **流式即异步生命周期**：字节重组 + 结束校验 + 断流补救 + 渲染副作用判空，四者缺一不可。
7. **分阶段交付**：滑动窗口先行，摘要压缩留待需要时；不过早优化，也不欠债。
8. **成本与安全意识**：token 预算控制费用；临时 Key 方案的泄露风险显式标注，上线走后端代理。
9. **每次改动 `npm run build` 验证**，遵守项目既有规范（CSS 变量、无外部 UI 库、类型不用 `any`）。

## 7. 关键参数与调优位

| 位置 | 参数 | 当前值 | 说明 |
| --- | --- | --- | --- |
