# 「背了吗」文件上传 → AI 生成记忆卡片 · 技术方案（交付给执行 Agent）

> 本文是可直接执行的技术方案。执行前**必须先通读** `CLAUDE.md` 和以下现有文件，理解现状后再动手：
> `src/types.ts`、`src/store.tsx`、`src/api.ts`、`src/App.tsx`、`src/utils/kimi.ts`、
> `src/components/LibraryPage.tsx`、`src/components/LearnPage.tsx`。
>
> 核心原则：**最大化复用现有能力，最小化侵入**。本产品是「文档 → 卡片生成器」，不是知识库；
> **不引入** RAG / 向量库 / 对象存储 / 消息队列 / 四级数据模型。
> **不修改** SRS 算法、云同步核心链路、`beile_ma_v3` localStorage 结构性字段、数据库 schema 结构。

---

## 0. 需求一句话

用户上传 `.md / .docx / .pdf` → 前端解析成纯文本 → 分块调 LLM 生成 `Question[]` →
写入现有 `store.custom` → 直接进入现有 SRS 学习流。原始文件不落地，只留元数据指纹用于去重。

---

## 1. 数据存储归属（必须严格遵守）

| 内容 | 存储位置 | 生命周期 |
|---|---|---|
| 原始文件（pdf/docx/md） | **仅浏览器内存**，用完即弃 | 解析完成即 GC，**绝不上传、绝不写 localStorage** |
| 文档元数据（名/hash/卡数/时间） | `store.custom` 无法承载，**复用 `user_progress`**，见 §4.3 | 长期 |
| 生成的卡片 `Question[]` | `store.custom` → 云端 `custom_json` + localStorage `beile_ma_v3` | 长期 |
| 卡片学习进度 | 现有 `store.cards`（**零改动**） | 长期 |

## 2. 关键边界约束（做之前先内化，违反会导致数据损坏）

### 2.1 卡片 id 冲突（最高优先级隐患）

- 现状：`Question.id: number`，内置题库 id 来自云端数据库**自增整数**（`api.ts:56` 直接用 `row.id`）。
  `allCards() = [...questions, ...store.custom]`（`store.tsx:403`），二者**共用 `store.cards[id]` 学习状态**。
- 风险：给生成卡片随手分配 `number` id 极易与云端题库 id 撞车 → 两张不同的卡共享同一份掌握度，状态串号。
- **约束（不改类型，用命名空间偏移）**：
  - **保持 `Question.id` 为 `number` 类型**（改成 string 会波及 `getCardState(id: number)`、
    `onPreviewCard(id: number)`、`buildQueue`、`rateCard` 等大量签名，代价过高，禁止）。
  - 生成卡片 id 采用**大偏移量命名空间**：`id = CUSTOM_ID_BASE + 单调递增序号`，
    其中 `CUSTOM_ID_BASE = 1_000_000_000`（10 亿）。云端题库自增 id 在可预见期内远达不到该量级。
  - 递增序号来源：`Math.max(已有 custom 卡片 id) + 1`，若无 custom 则从 `CUSTOM_ID_BASE + 1` 开始。
    **必须在 reducer 内、基于当前 `state.store.custom` 计算，禁止用 `Date.now()` 或 `Math.random()` 直接当 id**
    （避免并发/重复触发时碰撞）。
  - 常量定义在 `src/types.ts`，命名 `CUSTOM_ID_BASE`。

### 2.2 云同步 blob 膨胀

- 现状：`syncToCloud`（`api.ts:125`）每次全量序列化 `custom_json` 覆盖写。
- 约束：单次上传生成的卡片要**批量一次性写入**（一个 dispatch），触发**一次** debounce 同步（现有 2s debounce，
  `store.tsx:494`）。禁止每生成一张卡 dispatch 一次（会触发 N 次全量同步）。
- 软上限：`store.custom` 总卡片数超过 **2000** 时，UI 侧提示「本地卡片较多，建议清理」，不硬阻断。
  （数千级以上的拆表方案属于第三阶段，本次不做。）

### 2.3 分类 `cat` 与 Deck

- 现状：`LibraryPage` 用 `c.cat` 做筛选与分组（`LibraryPage.tsx:24-27,44`）。
- 约束：生成卡片**必须填 `cat`**（用文档名或 LLM 归纳的主题），这样天然进入现有题库筛选，**无需新建 Deck 表**。
  「Deck」在本期是逻辑概念 = `source.docId` 相同的一批卡片。

### 2.4 `Question` 字段完整性

- 现状：`LearnPage.renderAnswer`（`LearnPage.tsx:143`）直接读 `card.summary/a/pitfalls/interview[]`，
  `AIChatBot` 读 `card.q/a/pitfalls/interview`（`kimi.ts:54`）。
- 约束：生成的卡片**必须满足 `Question` 完整结构**，缺字段用安全默认值（见 §5.3），
  `keywords`/`interview` **必须是数组**（不是字符串——注意云端题库里它们是 JSON 字符串，
  但 `store.custom` 里存的是已解析的数组，见 `api.ts:62,64` 与 `MERGE_CLOUD` `store.tsx:326`）。

### 2.5 未登录 / 离线

- 现状：未登录也能用（localStorage 兜底），云同步仅在 `state.user` 存在时触发（`store.tsx:498`）。
- 约束：上传生成功能在**未登录状态也要可用**（卡片进 localStorage）。登录后由现有 `MERGE_CLOUD` 合并上云。
  **但 AI 调用需要网络**——未登录但有网也能生成；无网则报错（见 §6）。

## 3. 整体流程（MVP Pipeline）

```
[选择文件]
  → 校验（类型/大小）—— 失败：Toast，终止
  → 读取到内存 & 解析为纯文本 —— 失败：Toast，终止
  → 计算文本 hash —— 命中已导入：提示「该文档已导入过」，终止
  → 按标题/章节分块（带 token 兜底）
  → [逐块] 调 LLM 结构化生成 → JSON 容错解析 → 累积候选卡
  → 本地去重（题干归一化相似）
  → 批量写入 store.custom（一次 dispatch，带 source/deckId）+ 写入文档元数据
  → 提示「成功生成 N 张卡片」→ 可一键进入学习 / 去题库查看
```

每一步的异常处理见 §6，**任一步失败都不得产生半成品脏数据**（要么全部写入，要么什么都不写）。

## 4. 数据模型改动

### 4.1 扩展 `Question`（`src/types.ts`，全部可选，向前兼容）

在现有 `Question` interface 追加**可选**字段（老数据/云端题库无这些字段，读取时为 `undefined`，不影响现有逻辑）：

- `source?: { docId: string; docName: string }` —— 卡片来源文档，用于按文档管理/删除。
- `deckId?: string` —— 逻辑牌组 id，MVP 阶段 = `source.docId`。

**不新增** `tags`（留第二阶段）、**不新增** `DocumentChunk` / `Knowledge`。

### 4.2 新增文档元数据类型（`src/types.ts`）

```
interface DocMeta {
  docId: string        // uuid
  name: string         // 原始文件名
  type: string         // 'md' | 'docx' | 'pdf'
  hash: string         // 解析后纯文本的 SHA-256（去重用）
  cardCount: number    // 该文档生成的卡片数
  categories: string[] // 该文档最终使用的分类清单（5~12 个，去重收敛后），见 §11
  createdAt: number     // Date.now()
}
```

> 说明：这里仅描述结构，禁止在方案文档外输出实现代码；由执行 Agent 按项目 TS 规范落地。

### 4.3 `Store` 增加 `documents` 字段（`src/types.ts` + `src/store.tsx`）

- `Store` 追加：`documents?: DocMeta[]`（可选，兼容老数据，`loadLocal` 里 `d.documents || []`）。
- **持久化**：
  - localStorage：随现有 `saveLocal` 一起写入 `beile_ma_v3`（新增可选字段，**不破坏现有结构**，符合 CLAUDE.md 约束）。
  - 云端：复用 `user_progress`，新增列 `documents_json`。**遵循 `achievements_json` 的既有兼容范式**
    （`api.ts:107,121` + `store.tsx:327`）：老库无此列时读回 `undefined`，reducer 侧 `|| '[]'` 兜底。
    `syncToCloud` body 增加 `documents_json: JSON.stringify(store.documents || [])`。
    `loadFromCloud` 返回体增加可选 `documents_json`。`MERGE_CLOUD` 增加 documents 合并（按 docId 去重）。
- **注意**：`syncToCloud` 入参类型当前是 `{ cards; daily; custom; mode }`（`api.ts:126`），需扩展带上 documents。

### 4.4 新增 reducer action（`src/store.tsx`）

- `ADD_GENERATED_CARDS`：
  - 入参：`cards: Omit<Question,'id'>[]`（不含 id）、`doc: DocMeta`。
  - 行为（**在 reducer 内原子完成**）：
    1. 基于当前 `state.store.custom` 计算起始 id（§2.1）。
    2. 依次分配 id，回填 `source`/`deckId`。
    3. `custom = [...custom, ...新卡片]`，`documents = [...documents, doc]`。
    4. `saveLocal(store)`，返回新 state（触发 §2.2 的单次同步）。
- `REMOVE_DOCUMENT`（第二阶段可先留桩）：
  - 入参：`docId`。
  - 行为：从 `custom` 移除 `source.docId === docId` 的卡片；从 `documents` 移除该条；
    **同时清理 `store.cards` 中这些卡片 id 的学习状态**（避免残留孤儿状态）；`saveLocal`。

## 5. 各模块实现规范

### 5.1 文件解析 `src/utils/docParser.ts`（新增）

- 导出：`parseFile(file: File): Promise<{ text: string; type: string }>`。
- 分发：
  - `.md` / `.markdown` / `.txt`：`file.text()` 直接读。
  - `.docx`：用 `mammoth`（浏览器版 `extractRawText`）转纯文本。
    - **`.doc`（老二进制格式）不支持**——浏览器无可靠纯前端解析方案。校验阶段直接拒绝，Toast「暂不支持 .doc，请转存为 .docx」。
  - `.pdf`：用 `pdfjs-dist` 逐页 `getTextContent()` 拼接。
- 依赖引入：`mammoth`、`pdfjs-dist`。**必须核对与 React19/Vite8/TS6 的兼容性**，
  遵守 CLAUDE.md「不升级核心依赖」；pdfjs worker 用 Vite 的 `?url` / `import.meta.url` 方式加载，
  避免 worker 路径在 `base=/page/beile-ma-react/` 下 404（**部署 base 敏感，务必验证生产构建**）。
- **边界处理**：
  - 解析结果 `text.trim().length === 0` → 抛错「未能从文档中提取到文本（可能是扫描版 PDF/纯图片）」。
  - PDF 加密 / 解析异常 → 捕获并抛出可读错误。
  - 提取文本做基础清洗：合并 3+ 连续空行为 1 行、trim 行尾空白（不做激进清洗以免破坏结构）。

### 5.2 分块 `src/utils/chunker.ts`（新增）

- 导出：`chunk(text: string): Chunk[]`，`Chunk = { title: string; content: string }`。
- 策略（优先语义边界，token 兜底）：
  1. 优先按 Markdown 标题（`#`~`######`）切分，标题作为 `title`，标题到下个同级/更高级标题之间为 `content`。
  2. 无标题结构（纯 PDF/txt）时，按空行分段聚合。
  3. **token 兜底**：单块估算 token（复用 `kimi.ts:estimateTokens`，`store`/`utils` 均可 import）
     超过 `CHUNK_MAX_TOKENS = 1500` 时，在段落边界二次切分。
  4. 过小的相邻块（< `CHUNK_MIN_TOKENS = 200`）合并，避免产出零碎卡片。
- **边界处理**：
  - 空文本 → 返回 `[]`（上层据此提示「文档内容过少，未生成卡片」）。
  - 总块数上限 `MAX_CHUNKS = 40`（对应约几十页）。超出：**只处理前 40 块并提示「文档较大，仅处理前部分内容，建议拆分导入」**，不静默截断到用户无感。

### 5.3 卡片生成 `src/utils/cardGen.ts`（新增）

> **归类相关的完整规则见 §11「分类体系与掌握度统计」，本节只描述卡片字段生成主流程。**
> 关键前提：**`cat` 必须填"知识点分类"，绝不能填文档名**（填文档名会破坏现有分类筛选与分类掌握度，见 §11.0）。

- 导出：`generateCards(chunks, docName, categories, onProgress)`（异步）。
  `categories` 是 §11 阶段一先产出的**受控分类清单**（5~12 个），生成卡片时 LLM 只能从中选 `cat`。
  返回 `{ cards: Omit<Question,'id'>[]; failedChunks: number }`。
- **复用 `kimi.ts` 的能力，但新增非流式/结构化调用**：
  - 现有 `streamChat`（`kimi.ts:151`）为对话流式，不适合结构化 JSON。
  - **两种可选实现，任选其一并说明理由**：
    (a) 复用 `streamChat` 累积全文后 `JSON.parse`（省改动，但流式对 JSON 无收益）；
    (b) 在 `kimi.ts` 新增 `chatOnce(messages): Promise<string>`（非流式）。**推荐 (b)**，语义更清晰。
  - **禁止**新起一套 fetch，必须复用 `API_URL/API_KEY/MODEL` 常量与 `Authorization` 逻辑。
- Prompt 要求（强约束，写进 system/user）：
  - 输出**严格 JSON 数组**，每元素对应一个**原子知识点**卡片。
  - 字段对齐 `Question`：`q`(必填) / `summary` / `a`(必填,支持 Markdown) / `keywords`(string[]) /
    `pitfalls` / `interview`(string[]) / `diff`(1~3) / `cat`(**必填，只能取自传入的 `categories` 清单**)。
  - 约束「一卡一知识点、问题可独立作答、不要重复、难度用 diff 标注」。
  - 明确「只依据给定文本，不得编造」（沿用 `kimi.ts:buildSystemPrompt` 的防幻觉基调）。
- **每块单独一次调用**，`onProgress(done, total)` 回调驱动 UI 进度。
- **JSON 容错解析（关键，LLM 输出不稳定）**：
  - 先 strip ```` ```json ```` / ```` ``` ```` 代码围栏。
  - 尝试 `JSON.parse`；失败则用正则截取首个 `[` 到末个 `]` 再 parse。
  - 仍失败 → **该块跳过**（记为失败块），**不中断整体流程**，继续下一块。
  - 解析出的每张卡逐字段校验 + 兜底（见下），**丢弃 `q` 或 `a` 为空的卡**。
- **字段兜底与净化**：
  - `q`/`a` 缺失或空 → 丢弃该卡。
  - `summary`/`pitfalls` 缺失 → `''`。
  - `keywords`/`interview` 非数组 → `[]`；若是字符串尝试按行/逗号拆分。
  - `diff` 非 1~3 整数 → 默认 `2`（对齐 `api.ts:65` 默认值）。
  - `cat` → **必须映射回 `categories` 清单**：若 LLM 返回的 `cat` 不在清单内，就近归一化（见 §11.3）；
    仍无法匹配 → 归入清单里的兜底类「其他」（§11.3），**绝不填文档名**。
- **返回**：合法卡片数组 + `failedChunks`（供 UI 提示「N 段未能生成，已跳过」）。

### 5.4 去重

- **文档级（硬去重）**：解析后文本算 SHA-256（用 `crypto.subtle.digest`，注意需 https/localhost 环境，
  非安全上下文降级用简单字符串 hash 并注释说明）。与 `store.documents[].hash` 比对，命中即终止并提示。
- **卡片级（软去重，跨文档相同知识点）**：
  - 对新卡 `q` 做归一化（去标点/空白、转小写），与现有 `allCards()` 的 `q` 归一化集合比对；
    完全相同则跳过。MVP **只做完全相同去重**，不做相似度阈值（相似度留第二阶段，避免误杀）。

### 5.5 上传 UI `src/components/UploadSheet.tsx`（新增）

- 复用现有 Sheet/Overlay 样式（`base.css` 已有通用 Sheet/Overlay/Toast）。样式新增写 `src/styles/` 下
  对应模块文件末尾（如新建 `upload.css`），并在 `App.tsx` 顶部 import（遵守 CLAUDE.md CSS 规范）。
- 状态机：`idle → parsing → generating(带进度 done/total) → success(N 张) / error(可读信息)`。
- 入口位置：题库页 `LibraryPage` 顶部加「上传文档」按钮（不干扰首页主学习流，符合「学习入口合、管理入口分」）。
  首页**本期不加**上传入口，避免干扰核心流程。
- 生成成功后提供两个动作：「去学习」（可复用 `startLearn`，但注意 `startLearn` 在 `App.tsx`，
  需通过 props 透传或后续接线）、「在题库查看」（切到 library 页）。**接线方式与 `App.tsx` 现有
  `onStartLearn`/`onPreviewCard` 保持一致，禁止在组件内直接操作路由/状态。**
- **所有 Builder/AI 请求禁止在组件内直接 fetch**（CLAUDE.md 铁律）：AI 走 `utils/cardGen.ts`→`kimi.ts`；
  写状态走 `useApp()` 的 dispatch。

## 6. 异常处理清单（逐场景，必须全覆盖）

| 场景 | 处理 |
|---|---|
| 文件类型不支持（.doc/.epub/.html/其他） | 校验拒绝，Toast 明确说明支持的类型；`.doc` 单独提示转 `.docx` |
| 文件超大（> `MAX_FILE_SIZE = 5MB`） | 拒绝，Toast「文件过大，请控制在 5MB 内或拆分」 |
| 文本超长（解析后 > 约 3 万字 / 超 `MAX_CHUNKS`） | 只处理前 `MAX_CHUNKS` 块并**明确提示**部分处理 |
| 解析失败（损坏/加密/扫描版无文本层） | 捕获，Toast 可读错误，**不产生任何卡片** |
| 提取文本为空 | Toast「未提取到文本」，终止 |
| 文档 hash 命中已导入 | 提示「该文档已导入过（生成过 N 张卡）」，终止 |
| 无网络 / AI 请求失败（fetch 抛错 / 非 200） | Toast「AI 服务暂不可用，请稍后重试」；**已成功生成的块不写入**（要么全成要么不写，见下） |
| AI 返回非法 JSON（单块） | 跳过该块，计入 `failedChunks`，继续；最终若**全部块失败** → 报错终止，不写空数据 |
| AI 限流（429）| 捕获，指数退避重试**最多 2 次**；仍失败按 AI 请求失败处理 |
| 部分块成功、部分失败 | 写入成功卡片 + 提示「已生成 N 张，M 段未能解析已跳过」（这是可接受的部分成功） |
| 生成 0 张有效卡片 | 不写入，Toast「未能从该文档生成有效卡片」 |
| id 计算异常（custom 为空/id 非法） | 从 `CUSTOM_ID_BASE + 1` 起，防御性 `Number.isFinite` 校验 |
| crypto.subtle 不可用（非安全上下文） | 降级简单 hash，注释说明，不阻断 |
| 生成中用户关闭 Sheet / 切页 | `AbortController` 中断在途 AI 请求（参考 `AIChatBot` `abortRef` 用法），**不写半成品** |
| 云同步失败 | 沿用现有 debounce 静默重试机制，本地已 `saveLocal` 兜底，不额外报错弹窗 |

**原子性要求**：整份文档的卡片**在全部块处理完后一次性 dispatch 写入**；中途失败/中断则**一张都不写**
（除「部分块解析失败但至少一块成功」的可接受部分成功场景——此时写入所有成功卡片）。

## 7. 常量集中定义（`src/types.ts`）


## 11. 分类体系与掌握度统计（核心补充）

> 本节回答两个关键问题：**(1) 上传文档的内容如何分类；(2) 掌握度如何统计而不被多文档污染。**
> 决策：**分类粒度 = 每份文档 5~12 个大类（`DOC_CAT_MIN`~`DOC_CAT_MAX`）。**

### 11.0 为什么不能把 `cat` 填成文档名（先说清红线）

现有代码强依赖 `cat` 作为**知识点分类维度**：
- `LibraryPage.tsx:24` / `StatsPage.tsx:30`：`[...new Set(cards.map(c => c.cat))]` —— `cat` 直接就是
  题库筛选下拉项与「分类掌握度」的分组依据。
- `StatsPage.tsx:123`：`getCategoryIcon(cp.cat)` 按 `cat` 匹配图标。

若把 `cat` 填成文档名：① 每份文档只剩 1 个分类，看不出内部结构；② 分类图标退化为 fallback；
③ 文档一多，统计页分类列表变成一堆文档名，失去"知识点掌握度"的意义。
**因此 `cat` 恒为"知识点分类"，文档归属由 `source.docId` / `deckId` 承载（两级：文档→分类）。**

### 11.1 两阶段归类（防漂移 + 防爆炸，保证 5~12 类）

分类**不是逐块自由生成**，而是先定表、再打标：

- **阶段一 · 先产出受控分类清单**（`cardGen` 内新增一次 LLM 调用，在分块之后、逐块生成之前）：
  - 输入：文档的**标题大纲 / 各 chunk 的 title + 前若干字**（不必喂全文，省 token）；
    **同时喂入「题库现有分类清单」**（`[...new Set(allCards().map(c=>c.cat))]`）——见 §11.1.1。
  - 要求 LLM 产出一个 **`DOC_CAT_MIN`~`DOC_CAT_MAX` 个知识点大类**的 JSON 数组（覆盖全文、互斥、粒度均衡），
    并**优先复用**传入的现有分类，仅在现有分类覆盖不到时才新建。
  - 这份清单即 `DocMeta.categories`，也是「这份文档里有哪些分类」的**权威来源**。
- **阶段二 · 按清单打标**（逐块生成卡片时）：
  - 把 `categories` 清单随 prompt 传入，**强制每张卡的 `cat` 只能从清单里选**，不允许新造。
  - 好处：分类数量被框死在 5~12，"文档里有哪些分类"稳定可预期，不漂移、不爆炸。

> **关于分类数量下限（问题 1）**：`DOC_CAT_MIN=5` 只是"上限区间的参考下界"，**不是强制门槛**。
> 短文档/单一主题文档自然产出 2~4 个分类是正常的，**允许 < 5，绝不强行凑数**（硬凑会把完整概念切碎、
> 产出牵强分类）。真正需要严格约束的是**上限** `DOC_CAT_MAX=12`（防爆炸）。见 §11.5 异常表对应行。

### 11.1.1 复用题库现有分类（问题 2：全自动，不需用户确认）

需求：用户上传的文档内容若能归入题库**已有分类**（如已有「工具调用」，上传的 LLM 文档也讲工具调用），
应**并入现有分类**而非新造重名分类。策略：**全自动，无用户干预步骤**。

- **阶段一输入现有分类清单**：把 `allCards()` 去重后的现有 `cat` 列表随 prompt 传给 LLM，
  明确指令「**能对应到现有分类的知识点，必须直接复用现有分类名（一字不差）；只有现有分类都覆盖不到的，才新建**」。
- **阶段二/后处理二次校准**（防 LLM 没复用）：对最终每张卡的 `cat`，做归一化后与**现有分类**再比对一次，
  命中（含同义/大小写/空白差异）则**强制改写为现有分类的标准名**，确保不产生 `工具调用` / `工具调用 ` / `Tool Calling` 这类重复分类。
- **边界处理**：
  - 现有分类清单过大（题库分类很多）时，只传**与本文档大纲语义相关的候选**或全部传入但限制 token（清单本身通常几十条，token 可控）。
  - 复用后可能出现「一份文档的分类清单里，部分是现有分类、部分是新建」——**允许且正常**，`DocMeta.categories` 如实记录二者。
  - **重名即合并**：既然复用的是同名分类，题库筛选/统计天然把它们聚到一起，无需额外合并逻辑。
  - 全自动的代价：可能偶发误合并（把不完全相同的概念并入现有分类）。**当前阶段接受**（用户选择全自动）；
    若未来投诉多，再加"生成后可编辑分类"的轻量入口（留后续，不在本期）。

### 11.2 「文档里有哪些分类」的数据来源

| 需求 | 来源 |
|---|---|
| 某文档有哪些分类 | 直接读 `DocMeta.categories`（生成时固化，无需遍历卡片） |
| 某文档某分类有几张卡 | `custom.filter(c => c.source?.docId===docId && c.cat===x).length` |
| 全站有哪些分类 | 现有 `[...new Set(cards.map(c=>c.cat))]`（**代码零改动**，含内置题库+所有文档） |

### 11.3 分类归一化与孤儿合并（后处理兜底）

阶段二难免有个别卡的 `cat` 越界或近义漂移，生成完成后统一收敛：
- **越界映射**：卡的 `cat` 不在 `categories` 内 → 做归一化（去空白/大小写/常见同义）就近匹配清单项；
  仍无法匹配 → 归入 `DOC_CAT_FALLBACK`（「其他」），并把「其他」补进 `DocMeta.categories`。
- **孤儿合并**：统计该文档内每个分类的卡片数；仅含 1 张卡的"孤儿分类"数量超过
  `ORPHAN_CAT_MERGE_THRESHOLD` 时，把这些孤儿并入「其他」，避免统计页出现一堆 `1/1` 条目。
- 合并后**回写** `DocMeta.categories` 与各卡 `cat`，保证清单与卡片最终一致。

### 11.4 掌握度统计口径改造（`StatsPage.tsx`）

现状问题：全局掌握率 `mastered/cards.length`（`StatsPage.tsx:25-26`）把所有卡混算，
上传法律/医学文档后会**稀释「AI 面试」原本有意义的掌握率**；分类掌握度也会把跨领域分类平铺混排。

改造为**按来源（Deck）分区**：
- **顶部全局区**：只保留跨领域仍有意义的指标 —— 学习天数、总时长、总题数、连击。**移除/弱化"全局掌握率"混算**。
- **掌握率下沉为分区展示**，每个来源一块：
  - 「内置题库（AI 面试）」= `source` 为空的卡；
  - 每个上传文档 = 一个 `DocMeta`（用 `source.docId` 归组）。
  - 每块内：该来源整体掌握率 + 该来源内**按 `cat` 的分类掌握度**（复用现有进度条 UI，仅把数据源从
    "全量 cards" 改为 "该来源的 cards"）。
- 分组归属判定：`card.source?.docId ?? '__builtin__'`。
- **老数据兼容**：历史卡片无 `source` → 全部落入「内置题库」分区，行为与改造前一致，不丢不乱。

### 11.5 分类相关的异常处理（补充进 §6 的口径）

| 场景 | 处理 |
|---|---|
| 阶段一 LLM 未返回合法清单 / 清单为空 | 降级：用 chunk 的 `title` 去重作为分类清单；仍为空 → 用单一分类 `DOC_CAT_FALLBACK` |
| 清单数量 < `DOC_CAT_MIN` | 允许（短文档正常），不强行凑数 |
| 清单数量 > `DOC_CAT_MAX` | 截断到前 `DOC_CAT_MAX` 个，其余卡片 `cat` 就近映射或入「其他」 |
| 阶段二卡片 `cat` 越界 | 归一化就近匹配，失败入「其他」（§11.3），**绝不填文档名、绝不丢卡** |
| 同名分类跨文档重名（不同文档都有"概述"） | **允许重名**：统计按 `source.docId + cat` 双键分组，同名不冲突；全站筛选下重名分类会合并展示（可接受） |
| `getCategoryIcon(cat)` 未命中 | 需保证有**按 cat 字符串稳定派生**的兜底图标（而非全部同一 default），避免统计页图标雷同（改造 `categoryIcon.tsx` 的 fallback 分支） |

## 12. 生成卡片的排序：置顶展示 + 优先学习（问题 3）

需求：**AI 生成的卡片要排在题库最上面，且学习时也优先学新上传的卡。**
这同时影响「题库展示顺序」和「学习队列取题顺序」，二者都要改。

### 12.0 现状（改动前）

- 顺序源头：`allCards() = [...questions, ...store.custom]`（`store.tsx:403`）——**云端题库在前、custom 在后**。
- 题库页 `LibraryPage.tsx:21` 直接按此顺序渲染 → 生成卡片现在排在**最下面**（与需求相反）。
- 学习新题 `buildQueue('all')` 从 `pool`（同样源自 `allCards`）按分类 round-robin 取 `goal` 道（`App.tsx:129-150`）。

### 12.1 统一改动源头：`allCards()` 让 custom 在前，且按新→旧排序

- 将 `allCards()` 改为：`[...sortedCustom, ...questions]`，其中 `sortedCustom` 按**上传时间倒序**
  （最新文档的卡片最靠前）。排序键优先用 `DocMeta.createdAt`（同文档内保持生成顺序）；
  无 DocMeta 时降级用卡片 id 倒序（id 递增分配，见 §2.1，越大越新）。
- 这样**题库展示**（`LibraryPage` / `StatsPage` 分区内）与**学习取题**（`buildQueue`）同时获得"新卡在前"，
  一处改动、两个需求都满足，避免展示顺序与学习顺序割裂。
- **注意闭包依赖**：`allCards` 现为 `useCallback(..., [])`（`store.tsx:403`）依赖空数组、内部读 `stateRef.current`，
  排序逻辑写在内部即可，无需改依赖；但要确保排序**每次调用都基于最新 `store.custom / store.documents`**（读 ref，不缓存）。

### 12.2 保护「复习优先」这条核心红线（关键）

CLAUDE.md 铁律：**当天有到期复习题时，必须先学完复习题才解锁新题**。本次"优先学新卡"**只作用于"新题内部的先后顺序"**，
**绝不能**让新上传的卡插队到"复习队列"之前。约束：

- 只改 `buildQueue('all')`（新题池）与 `review-due` **无关**——`review-due` 队列（`App.tsx:111-116`）**不动**。
- 首页解锁判断 `canStartNew`（`store.tsx:490`）**不动**：复习没清空前，新题（含新上传卡）依旧不解锁。
- 即：**复习题 > 新上传卡 > 其他新题**，复习永远最优先。

### 12.3 新题内部排序与现有 round-robin 的关系

- 现有 `buildQueue('all')` 是**按分类 round-robin** 取题（`App.tsx:136-150`），并非简单顺序取。
- 落地"新卡优先"的推荐做法：**在 round-robin 之前，先把 `pool` 里属于最新文档的卡片整体前置**，
  或让 round-robin 的**分类遍历顺序**以"最新文档的分类"打头。二选一，执行 Agent 按可读性选，
  但必须保证：① 复习优先不破坏；② 仍是取 `goal` 道；③ 同一分类内保持稳定顺序（避免每次进入题目乱跳）。
- **边界**：若最新文档的新卡数量 ≥ `goal`，则今日新题可能全部来自该文档——**符合"优先学新卡"预期，允许**。
  用户若想学别的，可次日或通过题库单题预览进入。

### 12.4 排序相关异常/回归验证

- [ ] 上传文档后，题库页（对应来源分区内）该文档卡片出现在**最上方**，多份文档按上传时间新→旧排列。
- [ ] 学习"今日新题"时，**先出最新上传文档的卡**（在复习清空之后）。
- [ ] **复习优先未被破坏**：当天有到期复习题时，首页新题入口仍锁定，`review-due` 仍最先学。
- [ ] `allCards()` 改序后，**统计/成就/今日进度口径不受影响**（这些只用集合，不依赖顺序，需回归确认）。
- [ ] 题目在队列中**顺序稳定**，同一次学习内前后翻卡不乱跳。
